# Bot Web UI

React + Vite admin UI for the QQ bot. Backed by 14 HTTP backends mounted on the bot's **StaticServer** (default port 8888): file browser, reports, insights, zhihu, moments, qdrant explorer, daily stats, memory, cluster, tickets, lan, projects, output static.

## Two backend categories (read this if you're deploying a client bot)

The 14 WebUI backends naturally split into two categories — they have **different deployment requirements**, so each gets its own configurable base URL:

### Category 1: Shared content (host = single source of truth)

| Backend | Path | Why it's "shared" |
|---|---|---|
| File browser | `/api/files` | Shows host's `output/`, source code, etc. |
| Output static | `/output` | Files served by host's `output/` dir |
| Reports | `/api/reports` | Group / WeChat reports generated by host |
| Insights | `/api/insights` | WeChat article analysis on host |
| Zhihu | `/api/zhihu` | Zhihu feed cache pulled by host |
| Moments | `/api/moments` | WeChat moments ingested by host |
| Qdrant Explorer | `/api/qdrant` | Vector DB used by host |
| Daily Stats | `/api/stats` | Log statistics persisted by host |
| Memory Status | `/api/memory` | Memory facts persisted by host |

These represent **knowledge accumulated by ONE bot**. A client bot on another machine doesn't have any of this data, so it should cross-read it from the host.

**Configured by `VITE_STATIC_SERVER_BASE`**. Empty = same origin.

### Category 2: Per-machine runtime state

| Backend | Path | Why it's "per-machine" |
|---|---|---|
| Cluster | `/api/cluster` | Each bot runs its own Agent Cluster (jobs / tasks / workers / locks) |
| Tickets | `/api/tickets` | `tickets/` markdown directory at each bot's project root |
| LAN | `/api/lan` | Each bot has its own LAN role (host or client) |
| Projects | `/api/cluster/projects` | Each bot has its own ProjectRegistry pointing at LOCAL paths |

These **MUST** hit the local bot's own backend, never the host's. The default (empty = same origin) is what every deployment wants almost always — the WebUI is served by the same bot that owns the local state.

**Configured by `VITE_LOCAL_API_BASE`** (new in Phase 3). Empty = same origin.

## Deployment scenarios

### A) Host bot, single machine

Both env vars unset → everything is same-origin. Open `http://localhost:8888` (or `http://<lan-ip>:8888`) and everything works.

### B) Client bot reading host content (the common multi-machine case)

The client bot runs its own webui on its own machine, has its own cluster/tickets/lan, but pulls files / reports / etc from the host bot.

`webui/.env` on the client machine:

```
VITE_STATIC_SERVER_BASE=http://192.168.50.209:8889
# VITE_LOCAL_API_BASE unset → cluster/tickets/lan same-origin = client's own bot
```

Then `bun run build:admin` on the client and run the client bot. Open `http://<client-ip>:8888` → file/reports/etc come from host, cluster/tickets/lan come from client itself.

**CORS**: when `VITE_STATIC_SERVER_BASE` points at a different origin, the host's StaticServer must send `Access-Control-Allow-Origin` permitting the client's origin. The host's backends already set permissive CORS headers for the API endpoints (verify in `src/services/staticServer/`).

### C) Detached webui dev server (Vite on laptop, bot on NAS)

Both vars set to the same backend URL — both groups go to that single place:

```
VITE_STATIC_SERVER_BASE=http://192.168.1.100:8888
VITE_LOCAL_API_BASE=http://192.168.1.100:8888
```

### D) Shared content from host, local APIs default same-origin

If only `VITE_STATIC_SERVER_BASE` is set and `VITE_LOCAL_API_BASE` is unset, shared-content requests use the remote base while cluster/tickets/lan/projects use same-origin (the machine serving the WebUI).

## Two processes (dev mode)

| Process | Port | Role |
|--------|------|------|
| **Bot + StaticServer** | 8888 | Serves `/output/*` (disk files) and `/api/*` (all 14 backends). |
| **Vite dev server (this UI)** | 5173 | Serves the React app. Proxies `/api` and `/output` to 8888. |

Run both with one command from repo root:

```bash
bun run dev
```

In dev mode the Vite proxy forwards all `/api/*` requests to the local bot at port 8888 — both env vars stay unset because same-origin works through the proxy.
