# Claude Code 工作日志 - 2026-03-18

## Gemini tool use 多轮会话提前退出并回复空白内容

**问题描述**:
Gemini provider 在 tool use 多轮会话（generateWithTools loop）中提前退出并回复空白内容。

**原因分析**:
三个根本原因：
1. `GeminiProvider.generate()` 在 `options.messages` 存在时，把所有消息展平为纯文本，导致最后一条消息被重复（循环一次 + prompt 再追加一次），且 tool 消息（functionCall/functionResponse）以原始文本传入，Gemini API 无法识别 → 第二轮可能返回空文本 + 无 functionCall → loop 提前退出。
2. `candidate.content.parts` 中的 functionCall 被遗漏（只检查了 `response.functionCalls` SDK 属性，未检查 parts 作为 fallback）。
3. `LLMService.toolUseProviders` 默认不包含 `gemini`，导致 gemini 被路由到其他 provider 处理 tool use，而不是自己处理。

**解决方案**:
- 新增 `mapChatMessagesToGeminiContents()` 将 ChatMessage[] 转换为 Gemini 原生 Content[] 格式：
  - system → `systemInstruction` config 字段
  - assistant + tool_calls → `{role:'model', parts:[{functionCall:{name,args}}]}`
  - tool role → `{role:'user', parts:[{functionResponse:{name,response}}]}`
- `generate()` 在 `options.messages` 存在时走 multi-turn 路径（不再追加 prompt）
- 添加从 `candidate.content.parts` 检测 functionCall 的 fallback
- `toolUseProviders` 默认增加 `gemini`

**相关文件**:
- `src/ai/providers/GeminiProvider.ts` - 主要修改
- `src/ai/services/LLMService.ts` - toolUseProviders 默认值
- `src/conversation/ConversationInitializer.ts` - toolUseProviders 默认值

## Gemini tool use 多轮会话 toolCallId 缺失导致卡片渲染失败

**问题描述**:
Gemini 通过 providerName 触发回复时，即使调用了 `format_as_card` tool，最终仍以文本（sendasforward）模式发送，而非卡片图片。

**原因分析**:
Gemini API 不返回 `toolCallId`（这是 OpenAI 的概念），导致 `LLMService.generateWithTools()` 进入 `else` 分支：
1. 推入 `{ role: 'assistant', content: '' }` — 无 `tool_calls` 属性
2. 推入 `{ role: 'user', content: 'Tool result for ...' }` — 纯文本

当 `mapChatMessagesToGeminiContents()` 处理这些消息时：
- 空 assistant 消息被跳过（parts.length === 0）
- tool 结果变成普通 user 消息
- 产生两个连续的 user 消息，且无 functionCall/functionResponse 对
- Gemini 不理解这是 tool 响应，生成的不是有效卡片 JSON → `tryRenderCardReply` 失败 → 回退到纯文本

**解决方案**:
- 移除 `toolCallId` 存在性判断分支，始终使用结构化的 `tool_calls` 格式
- 当 provider 不返回 `toolCallId` 时，生成合成 ID（`call_{round}_{timestamp}`）
- 这样 `mapChatMessagesToGeminiContents()` 可以正确转换为 `functionCall` / `functionResponse` 格式
- 同时更新 `config.example.jsonc` 将 gemini 加入 `toolUseProviders`

**相关文件**:
- `src/ai/services/LLMService.ts` - generateWithTools 的 tool call 消息构建
- `config.example.jsonc` - toolUseProviders 配置

## CardRenderer 外部 CDN 加载超时导致所有卡片渲染失败

**问题描述**:
所有卡片渲染（card render）均失败，报错 "Navigation timeout of 30000 ms exceeded"。LLM 正确生成了 card JSON 并通过 `format_as_card` tool 调用，但最终以 `sendAsForward`（纯文本/JSON）方式发送，而非图片。当天 8+ 次卡片渲染全部失败。

**原因分析**:
`CardRenderer.ts` 的 HTML 模板中引用了外部 CDN 脚本：
```html
<script src="https://unpkg.com/twemoji@latest/dist/twemoji.min.js" crossorigin="anonymous"></script>
```
同时 `page.setContent()` 使用 `waitUntil: 'networkidle0'`，要求所有网络请求完成。当 unpkg.com 在当前网络环境下不可达时，30 秒超时触发，渲染失败。

fallback 机制尝试用 DeepSeek 重新生成 card JSON 后再次渲染，但同样因 CDN 超时失败，最终回退为纯文本。

**解决方案**:
- 移除外部 twemoji CDN script 标签（emoji 使用系统原生渲染）
- `waitUntil` 从 `networkidle0` 改为 `domcontentloaded`（所有资源已内联，无需等待网络）
- 移除 twemoji 等待/解析逻辑
- 保留 `document.fonts.ready` 等待（确保系统字体加载完成）

**相关文件**:
- `src/services/card/CardRenderer.ts` - HTML 模板和渲染逻辑

## Gemini thinking model `thought_signature` 缺失导致 tool use 400 错误

**问题描述**:
Gemini provider 在多轮 tool use 会话中，每一轮都返回 400 错误：`Function call is missing a thought_signature in functionCall parts`。Gemini 始终失败并 fallback 到 DeepSeek，最终因 max rounds 达到上限，DSML 被 strip 后返回空内容。

**原因分析**:
Gemini thinking model（如 gemini-2.5-flash）在返回 functionCall 时会附带 `thoughtSignature` 字段（在 part 对象上）。API 要求在下一轮发送历史消息时，functionCall part 必须包含原始的 `thoughtSignature`。

当前代码流程中 `thoughtSignature` 在多处被丢弃：
1. `generateContentText()` 从 response part 提取 functionCall 时未捕获 `thoughtSignature`
2. `LLMService.generateWithTools()` 构建 `tool_calls` 消息时未携带
3. `mapChatMessagesToGeminiContents()` 重建 functionCall part 时未包含

**解决方案**:
- `ChatMessageToolCall` 增加可选字段 `thought_signature`
- `AIGenerateResponse` 增加可选字段 `thoughtSignature`
- `GeminiProvider.generateContentText()` 从 response part 捕获 `thoughtSignature`
- `LLMService.generateWithTools()` 将 `thoughtSignature` 写入 `tool_calls[].thought_signature`
- `mapChatMessagesToGeminiContents()` 将 `thought_signature` 回填到 functionCall part

**相关文件**:
- `src/ai/types.ts` - ChatMessageToolCall, AIGenerateResponse 类型
- `src/ai/providers/GeminiProvider.ts` - 捕获和回填 thoughtSignature
- `src/ai/services/LLMService.ts` - 透传 thoughtSignature

## max rounds 达到上限后 DSML strip 导致空回复

**问题描述**:
当 provider（如豆包）不支持 tool use 时，fallback 到 DeepSeek 处理。达到 max tool rounds 后，强制最终回复仍包含 DSML 格式的 tool call，被 strip 后 `responseLength=0`，用户收到空回复。

**原因分析**:
`generateWithTools()` 达到 max rounds 后调用 `generateMessages()` 生成最终回复，但不传入 tools。DeepSeek 仍然尝试通过 DSML 文本格式调用工具，被 `stripDSML()` 移除后回复为空。

**解决方案**:
在 max rounds 强制最终回复前，追加一条 user 消息明确告知模型已用完所有 tool 调用次数，要求直接给出文本回答。

**相关文件**:
- `src/ai/services/LLMService.ts` - generateWithTools max rounds 处理
