### 技术决策

1. **独立数据库模式**: 参照 WeChat 服务，使用独立 `data/zhihu.db`，不依赖核心 DatabaseManager
2. **Plugin 模式**: 注册为 `zhihuFeed` 插件，配置在 `plugins.list` 中，符合项目 plugin 架构
3. **直接 node-cron**: 使用 `node-cron` 而非 AgendaService，因为 poll/digest 是确定性操作，不需要 LLM 驱动的 AgentLoop
4. **Verb 过滤**: 实际 API 返回的 verb 与文档描述有差异（如 `MEMBER_VOTEUP_ANSWER` vs `ANSWER_VOTE_UP`），已更新默认 filter

### 遇到的问题

1. **`stripHtml` 参数类型**: 知乎 API 的 `excerpt`/`content` 有时不是 string（可能是 object），导致 `.replace()` 失败。修复：将 `stripHtml` 入参改为 `unknown` 类型并增加类型检查。
2. **`as const` 与 mutable 类型不兼容**: `DEFAULT_ZHIHU_CONFIG` 使用 `as const` 后，其 `verbFilter` 变为 readonly，与 `ZhihuConfig.verbFilter: string[]` 不兼容。修复：在 plugin 中手动展开默认值而非使用 intersection type。

## execute_code Tool 实现

### 任务描述

实现 `execute_code` 工具，允许 LLM 编写并执行 JavaScript 代码。执行环境注入其他工具作为全局函数，支持 async/await、console 输出捕获和超时保护。

### 实现内容

#### 新增文件

| 文件 | 说明 |
|------|------|
| `src/tools/executors/executeCode/types.ts` | 沙箱配置、执行结果、全局上下文等类型定义 |
| `src/tools/executors/executeCode/SandboxContext.ts` | 构建执行环境：包装其他工具为全局函数、捕获 console 输出、注入安全的 JS 标准库 |
| `src/tools/executors/executeCode/CodeSandbox.ts` | 代码执行引擎：AsyncFunction 构造、超时控制、返回值序列化、错误格式化 |
| `src/tools/executors/executeCode/ExecuteCodeToolExecutor.ts` | 主 Executor，`@Tool()` 装饰器注册，协调 SandboxContext 和 CodeSandbox |
| `src/tools/executors/executeCode/index.ts` | Barrel export |

#### 修改文件

| 文件 | 说明 |
|------|------|
| `src/tools/executors/index.ts` | 添加 `ExecuteCodeToolExecutor` 导出 |

### 架构设计

```
ExecuteCodeToolExecutor (入口)
├── SandboxContext (环境构建)
│   ├── buildToolFunctions() — 将其他 tool executors 包装为 async 函数
│   ├── buildConsole() — 捕获 console.log/warn/error/info
│   └── 注入安全 JS 标准库 (fetch, JSON, Math, Date, etc.)
└── CodeSandbox (执行引擎)
    ├── buildFunction() — AsyncFunction 构造，globals 作为命名参数注入
    ├── executeWithTimeout() — Promise.race 超时保护
    └── serializeReturnValue() — 返回值安全序列化
```

### 关键设计决策

1. **AsyncFunction 构造器方式**：将代码包装为 AsyncFunction，globals 作为命名参数传入，用户代码可直接引用 `tools`、`console`、`fetch` 等
2. **工具递归防护**：`execute_code` 自身不暴露给沙箱中的 `tools` 对象，避免递归调用
3. **内部工具排除**：visibility 为 `internal` 的工具不暴露给沙箱
4. **自动 return 包装**：如果代码不含显式 `return`，自动包装为 async IIFE 返回最后表达式的值
5. **超时保护**：默认 10s，最大 30s，通过 Promise 竞争实现
6. **输出限制**：最大 8000 字符输出、100 条 console.log

- typecheck: ✓
- lint: ✓（新文件无 error/warning）
- smoke-test: ✓（tool 成功注册为第 22 个工具）

## 微信日报功能修复（wechat report）

### 问题描述

微信日报功能失效：从 QQ 私聊或 agenda 定时任务触发时，生成的日报内容为"单聊模式下无法获取消息统计及互动分析数据"，未实际分析微信数据库内容。WebUI 报告链接也无效。

### 原因分析

**根本原因**：5 个核心问题叠加导致功能完全失效。

1. **工具可见性错误（核心）**：所有 6 个原始微信工具（`wechat_stats`、`wechat_report`、`wechat_digest`、`wechat_search`、`wechat_group_summary`、`wechat_article_summary`）的 `visibility` 设为 `['internal']`，导致它们既不在 `reply` 作用域（AgentLoop 使用）也不在 `subagent` 作用域（SubAgentExecutor 使用）中。LLM 没有任何微信工具可调用，只能凭自身知识生成文本。

2. **AgentLoop 工具作用域不足**：`AgentLoop.generateReply()` 使用 `getReplyToolDefinitions()` 仅获取 reply 作用域的工具。即使修复微信工具的 visibility 为 `['subagent']`，AgentLoop 仍然看不到。

3. **Schedule 元数据缺失**：`data/agenda/schedule.md` 中"微信日报生成"任务缺少 `- 用户: 528992419` 元数据字段，可能导致任务被跳过。

4. **SubAgentRolePresets 工具列表不完整**：`wechat_report` 预设缺少较新的工具（`wechat_article_insights`、`wechat_article_rag`、`wechat_digest`）。

5. **报告 URL 路径错误**：`getReportUrl()` 生成的 URL 包含 `/files/` 前缀（`/files/#/report/{id}`），但 WebUI hash 路由从根开始（`/#/report/{id}`）。

### 解决方案

| 文件 | 操作 | 说明 |
| --- | --- | --- |
| 6 个 wechat executor | 修改 | `visibility: ['internal']` → `['subagent']` |
| `src/agenda/AgentLoop.ts` | 修改 | 新增 `getAgendaToolDefinitions()` 合并 reply + subagent 作用域工具 |
| `data/agenda/schedule.md` | 修改 | 添加 `- 用户: 528992419` 元数据，优化 intent 措辞 |
| `SubAgentRolePresets.ts` | 修改 | 添加 `wechat_article_insights`、`wechat_article_rag`、`wechat_digest` |
| `WechatReportService.ts` | 修改 | URL 格式 `/files/#/report/{id}` → `/#/report/{id}` |

### 关键架构洞察

- 工具可见性（visibility）控制了工具在不同上下文中的可用性：`reply`（LLM 回复流）、`subagent`（子代理执行）、`internal`（仅内部调用，LLM 不可见）
- AgentLoop 是系统级执行器，需要比普通回复更广的工具访问范围（reply + subagent）
- 三个较新的微信工具（`wechat_analyze_articles`、`wechat_article_rag`、`wechat_article_insights`）没有设置 visibility，默认为 `['reply', 'subagent']`，不受此问题影响
