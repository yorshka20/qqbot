在 `wechat_oa_articles` 表中新增 `accountCategory TEXT NOT NULL DEFAULT ''`：
- 通过 ALTER TABLE 迁移添加（存量数据默认为空字符串）

### 关于 category 字段的调研结论

- WeChat OA push XML: `<mmreader><category>` 是包含 `<item>` 的容器元素，不是分类标签
- Chat article XML: 字段包括 title/url/des/thumburl/appid/sourcedisplayname/appname，无 category
- 现有 `wechat_article_insights.categoryTags` 是文章内容级别的 LLM 分析标签（如 `["科技","AI"]`）
- 新增的 `accountCategory` 是账号级别的分类（如"新闻"/"财经"），需通过 LLM 或手动赋值
- 存量数据可后续通过脚本基于已有 insights 的 categoryTags 聚合推断

## Provider Trigger 失败修复（Qwen3 thinking model）

**任务**: 通过 providerName 前缀（如 "claude，..."）触发特定 provider 回复失败

### 问题追踪

用户消息: `claude，测试一下你的功能，告诉我目前日美汇率及日中汇率是多少`

日志追踪（`logs/2026-03-22/2026-03-22-01-05-54.log:32166`）：

1. `MessageTriggerPlugin` 检测到 provider name 前缀 "claude"
2. 调用 `checkPrefixInvitation()` 进行 LLM 确认（使用 `qwen/qwen3-32b` via Groq）
3. `parseLlmTrueFalse()` 返回 `null` — LLM 响应不是纯 "true"/"false"
4. 消息被标记为 `postProcessOnly`，走 no-reply 路径

### 根因分析

`qwen/qwen3-32b` 是 thinking model，响应包含 `<think>...</think>` 标签。流程：

1. `LLMService.generateLite` 设置 `reasoningEffort: 'minimal'`
2. `GroqProvider.generateLite` 覆盖为 `reasoningEffort: 'none'`
3. **但 `GroqProvider.generate` 没有将 `reasoningEffort` 映射到 Groq API 的 `reasoning_format` 参数**
4. Qwen3 返回 `<think>...思考过程...</think>\ntrue`
5. `parseLlmTrueFalse` 只检查第一行 → 看到 `<think>` → 返回 `null`

### 解决方案

**两层修复**:

1. **`parseLlmTrueFalse`** (`src/ai/utils/llmJsonExtract.ts`): 解析前先 strip `<think>...</think>` 块（防御性）
2. **`GroqProvider.generate`** (`src/ai/providers/GroqProvider.ts`): 当 `reasoningEffort === 'none'` 时，传递 `reasoning_format: 'hidden'` 给 Groq API

### 修改文件

| 文件 | 修改内容 |
|------|----------|
| `src/ai/utils/llmJsonExtract.ts` | `parseLlmTrueFalse` 添加 `<think>` 标签清理 |
| `src/ai/providers/GroqProvider.ts` | `generate` 方法添加 `reasoning_format: 'hidden'` 映射 |
