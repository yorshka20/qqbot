# Claude Code Workflow Guide

本文档定义了 Claude Code 执行任务的标准工作流程。无论任务类型如何，都应遵循此流程以确保质量和一致性。

---

## 工作流程概览

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   RECEIVE   │───▶│   ANALYZE   │───▶│    PLAN     │───▶│   EXECUTE   │───▶│   VERIFY    │
│   接收任务   │    │   分析理解   │    │   规划拆分   │    │   逐步实现   │    │   验收交付   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

每个阶段都有明确的输入、输出和检查点。

---

## Phase 1: RECEIVE - 接收任务

### 目标
理解任务来源和基本要求。

### 输入
- 用户请求（自然语言描述）
- 相关文档路径（如 plan 文档）
- 上下文信息（项目、分支等）

### 动作
1. **确认工作目录**
   ```bash
   pwd
   git status
   git branch --show-current
   ```

2. **阅读必要文档**
   - `CLAUDE.md` - 项目开发规范
   - `template/LEARNINGS.md` - 项目知识库（架构、模式、陷阱）
   - 如有指定的 plan 文档，一并阅读

3. **阅读相关代码**（如有指定）
   - 读取相关代码文件
   - 参考 LEARNINGS.md 中记录的代码模式

4. **确认任务范围**
   - 明确需要做什么
   - 明确不需要做什么
   - 识别潜在风险或依赖

### 输出
- 对任务的理解摘要
- 需要进一步澄清的问题（如有）

### 检查点
- [ ] 工作目录正确
- [ ] 相关文档已阅读
- [ ] 任务范围已确认

---

## Phase 2: ANALYZE - 分析理解

### 目标
深入理解现有代码和实现要求。

### 动作
1. **探索相关代码**
   - 找到需要修改的文件
   - 理解现有实现模式
   - 识别可复用的代码

2. **理解依赖关系**
   - 分析文件间的依赖
   - 确认需要新增的依赖
   - 检查是否有破坏性变更

3. **识别参考实现**
   - 找到类似功能的现有实现
   - 理解项目的代码风格
   - 确定要遵循的模式

### 输出
```markdown
## 分析结果

### 涉及文件
- `path/to/file1.ts` - 需要修改（原因）
- `path/to/file2.ts` - 需要新建
- `path/to/file3.ts` - 参考实现

### 依赖关系
- 依赖于: xxx
- 被依赖于: xxx

### 参考模式
参考 `path/to/reference.ts` 的实现方式

### 风险点
- 风险1: xxx
- 风险2: xxx
```

### 检查点
- [ ] 相关文件已定位
- [ ] 依赖关系已理清
- [ ] 实现模式已确定

---

## Phase 3: PLAN - 规划拆分

### 目标
将任务拆分为可执行的子任务，每个子任务独立且可验证。

### 动作
1. **拆分子任务**
   - 每个子任务应该独立可完成
   - 子任务粒度适中（10-30分钟可完成）
   - 子任务有明确的完成标准

2. **确定执行顺序**
   - 考虑依赖关系
   - 基础设施优先
   - 核心功能次之
   - 集成测试最后

3. **定义验收标准**
   - 每个子任务的验收条件
   - 整体任务的验收条件

### 输出格式
```markdown
## 实现计划

### 子任务列表

#### Task 1: [任务名称]
- **目标**: 简述要完成什么
- **涉及文件**: 列出需要修改/新建的文件
- **实现要点**:
  1. 具体步骤1
  2. 具体步骤2
- **验收标准**: 如何确认完成

#### Task 2: [任务名称]
...

### 执行顺序
1 → 2 → 3 → 4

### 整体验收标准
- [ ] 标准1
- [ ] 标准2
- [ ] typecheck 通过
- [ ] lint 通过
- [ ] 测试通过（如适用）
```

### 检查点
- [ ] 子任务划分合理
- [ ] 执行顺序明确
- [ ] 验收标准清晰

---

## Phase 4: EXECUTE - 逐步实现

### 目标
按计划逐个完成子任务。

### 工作循环
对于每个子任务：

```
┌──────────────────────────────────────────────────────┐
│  1. 声明当前任务                                      │
│  2. 实现代码                                         │
│  3. 本地验证（typecheck/lint）                        │
│  4. 确认子任务完成                                    │
│  5. 如有必要，提交（或积累后统一提交）                   │
└──────────────────────────────────────────────────────┘
```

### 代码规范
1. **风格一致性**
   - 遵循项目现有代码风格
   - 使用项目配置的 formatter
   - 参考类似文件的实现

2. **最小改动原则**
   - 只修改必要的代码
   - 不做无关的重构
   - 不添加额外的功能

3. **错误处理**
   - 添加必要的错误处理
   - 使用项目的错误处理模式
   - 日志记录关键信息

### Git 规范
1. **Commit Message 格式**
   ```
   <type>(<scope>): <description>

   [optional body]

   Co-Authored-By: Claude <noreply@anthropic.com>
   ```

2. **Type 类型**
   - `feat`: 新功能
   - `fix`: 修复 bug
   - `refactor`: 重构（不改变功能）
   - `docs`: 文档更新
   - `test`: 测试相关
   - `chore`: 构建/工具相关

3. **提交时机**
   - 完成一个独立的功能点
   - 代码处于可工作状态
   - 已通过 typecheck 和 lint

### 输出
每完成一个子任务，输出：
```markdown
### Task N 完成

**实现内容**:
- 修改了 `file1.ts`: 添加了 xxx
- 新建了 `file2.ts`: 实现了 xxx

**验证结果**:
- typecheck: ✓
- lint: ✓

**下一步**: Task N+1
```

### 检查点（每个子任务）
- [ ] 代码实现完成
- [ ] typecheck 通过
- [ ] lint 通过
- [ ] 符合项目代码风格

---

## Phase 5: VERIFY - 验收交付

### 目标
确保整体实现符合要求，准备交付，并更新项目知识库。

### 动作
1. **运行完整检查**
   ```bash
   bun run typecheck
   bun run lint
   bun test  # 如有测试
   bun run build  # 如需要
   ```

2. **回顾实现**
   - 检查所有修改的文件
   - 确认没有遗漏
   - 确认没有多余的修改

3. **更新知识库** `template/LEARNINGS.md`
   - 如果发现了新的架构知识，更新"架构概览"
   - 如果使用了可复用的代码模式，更新"代码模式"
   - 如果踩了坑并解决了，添加到"常见陷阱"或"已解决的问题"
   - 如果发现待改进项，添加到"待改进项"

4. **准备总结**
   - 列出所有变更
   - 说明实现的关键决策
   - 标注任何需要注意的事项

### 输出格式
```markdown
## 任务完成报告

### 变更摘要
- 新增 N 个文件
- 修改 N 个文件
- 删除 N 个文件

### 文件清单
| 文件 | 操作 | 说明 |
|------|------|------|
| path/to/file1.ts | 新增 | 实现了 xxx |
| path/to/file2.ts | 修改 | 添加了 xxx |

### 验证结果
- [x] typecheck 通过
- [x] lint 通过
- [x] build 通过
- [ ] 测试通过（无测试/N个测试通过）

### 后续建议（如有）
- 建议1
- 建议2

### Commit/PR 信息
- Commit: `abc1234`
- Branch: `feat/xxx`
- PR: #123（如已创建）
```

### 检查点
- [ ] 所有检查通过
- [ ] 变更已提交
- [ ] 报告已输出

## 特殊场景处理

### 遇到阻塞问题
1. 清晰描述问题
2. 说明已尝试的解决方案
3. 提出需要的帮助或决策
4. 暂停等待反馈

### 发现 Plan 有问题
1. 指出 Plan 中的问题
2. 提出修正建议
3. 等待确认后再继续

### 任务范围扩大
1. 识别出超出原始范围的内容
2. 报告发现
3. 建议是否纳入当前任务或另开任务

### 需要用户确认
1. 清晰描述决策点
2. 给出选项和建议
3. 等待用户反馈

---

## 通用原则

### DO
- 先理解再动手
- 小步快跑，频繁验证
- 保持代码风格一致
- 遵循最小改动原则
- 及时报告进度和问题

### DON'T
- 不做无关的重构
- 不添加额外功能
- 不跳过验证步骤
- 不忽略错误信息
- 不假设用户意图

## 快速参考

### 常用命令
```bash
# 代码质量检查
bun run typecheck
bun run lint
bun run lint:fix

# Git 操作
git status
git diff
git add -A
git commit -m "type: description"

# 项目构建
bun run build
bun test
```

### 项目特定规范
参考项目根目录的 `CLAUDE.md` 文件。
