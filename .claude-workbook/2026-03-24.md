## 群聊汇报功能 Bug 修复（续）- Puppeteer 渲染超时

**任务**: 上一轮修复后重新触发 `/report`，SubAgent 成功调用 deepseek 并完成分析，但 `render_group_report` 工具执行时 Puppeteer 报 "Navigation timeout of 30000 ms exceeded"，图片未能渲染发送。

**问题分析** (通过 logs/2026-03-24/2026-03-24-02-03-20.log):
- SubAgent 正确使用了 deepseek provider，工具调用链正常完成
- `render_group_report` 被调用两次，均在 `page.setContent()` 阶段超时
- 根因: `waitUntil: 'networkidle0'` 要求所有网络请求完成后再继续，但 QQ 头像 CDN (`q1.qlogo.cn`) 响应慢/不可达，导致 30s 默认超时
- 头像加载实际已在后续 `page.evaluate` 中通过 `Promise.allSettled` + 5s 单图超时独立处理，`networkidle0` 是冗余且有害的等待

**修复方案**:
- `GroupReportRenderer.ts`: 将 `waitUntil` 从 `'networkidle0'` 改为 `'domcontentloaded'`，仅等待 DOM 解析完成，外部资源加载由后续代码处理

**涉及文件**:
- `src/services/groupReport/GroupReportRenderer.ts` - 修复 Puppeteer 页面加载策略

## Provider 显式选择被健康检查静默替换

**任务**: 用户通过 `claude xxx` / `gemini xxx` 等前缀显式指定 provider，但实际回复总是 fallback 到 DeepSeek。

**问题分析**:
- `ProviderRouter` 正确地将 `claude` 映射为 `anthropic`，设置 `ctx.selectedProviderName`
- `GenerationStage` 正确地将 `selectedProviderName` 传递给 `LLMService.generateWithTools()`
- **根因**: `LLMService.getAvailableProvider()` 在找到请求的 provider 后，会无条件执行健康检查（lines 159-170）。如果 provider 之前曾因瞬态错误被标记为 unhealthy，健康检查会静默将其替换为第一个 healthy fallback（deepseek）
- 调用者无法区分返回的 provider 是请求的还是被替换的

**修复方案**:
- 在 `getAvailableProvider()` 中添加 `explicitlyRequested` 标志
- 当 `providerName` 参数成功匹配到可用 provider 时，设置 `explicitlyRequested = true`
- 健康检查 fallback 仅在非显式请求时触发（session/default provider）
- 显式请求的 provider 即使 unhealthy 也会尝试执行，运行时错误由 `GenerationStage.generateWithRetry()` 处理

**涉及文件**:
- `src/ai/services/LLMService.ts` - `getAvailableProvider()` 方法
